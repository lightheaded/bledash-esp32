# MVP: ESP32‑C3 MINI + 0.42″ OLED reading Alpicool K25 + EcoFlow River 2 via BLE

**Date:** 2026‑07‑08
**Status:** proposed
**Owner:** Tom
**Target hardware:** MINI ESP32‑C3 dev board (RISC‑V, BLE 5.0, onboard 72×40 SSD1306 OLED, USB‑C), €3.34, already in inventory.
**Target devices:** Alpicool K25 (car fridge), EcoFlow River 2 (portable power station).
**Power source:** car USB port (5 V, always‑on while the car is on) — **no battery, no deep sleep, no e‑ink**. This is a change from the earlier "off‑grid car dashboard" idea; that variant is deferred to a v2 plan.

## Goals

A tiny always‑on display that lives in the car showing:
- Alpicool: target temp, actual temp, on/off, battery‑protection mode.
- EcoFlow: battery %, input W, output W, remaining time.

Rotating pages on the 72×40 OLED, one device per page, ~3 seconds each. Poll each device once per minute.

## Non‑goals for this plan

- Battery‑powered operation (deferred to v2 — see the LOLIN S3 Mini + e‑ink shield plan when it's written).
- WiFi / MQTT / cloud (BLE only for MVP).
- OTA updates (flash via USB is fine for now).
- Web UI / config over SoftAP (device MACs compiled in via `config.h`).
- Multi‑fridge or multi‑battery support (one of each).

## Hardware

### The board

- **MINI ESP32‑C3 Development Board with 0.42" OLED** (AliExpress €3.34, already in inventory).
- ESP32‑C3 (RISC‑V single core, 160 MHz), 4 MB flash, WiFi 4, **BLE 5.0**.
- **Onboard 72×40 SSD1306 OLED**, I²C on `SDA=GPIO5`, `SCL=GPIO6` (verify against silkscreen — some clones vary; document the actual pins after unboxing).
- USB Type‑C. Ceramic antenna. No PSRAM, no SD, no battery‑management.

### The car cable

Short USB‑A → USB‑C from the car's dashboard USB port. Nothing fancy. Length TBD once we know where the board sits.

### Physical mount

Out of scope for the firmware plan. A 3D‑printed cradle is likely; that's for a follow‑up.

## Constraints imposed by the hardware

1. **Single core.** ESP32‑C3 is one RISC‑V core. Cannot literally do "concurrent BLE + display render" in parallel like an S3 could. Must serialize: poll device A → poll device B → render → sleep. This is fine at a 60‑second cadence.
2. **Tiny display.** 72×40 pixels is roughly 2 lines of readable text and one large number, *or* one big number and a label. UI must be extremely reductive. One device per page is the right unit.
3. **BLE range.** ESP32‑C3 with ceramic antenna, mounted in a metal car, reading a fridge in the boot and a power station on the floor behind the driver — range will be marginal. Plan a test drive early to confirm actual RSSI at those distances **before** investing in UI polish.
4. **No hardware RTC.** Not needed — always‑on USB power means we don't sleep; the ESP32‑C3's internal timekeeping is sufficient for scheduling the 60‑second poll.

### BLE library choice

**NimBLE-Arduino**, not the built‑in ESP32 BLE library. NimBLE uses ~50% less RAM/flash, is actively maintained, and is standard for ESP32‑C3 projects because the classic ESP32 BLE stack is heavier than what a C3 comfortably runs. This is the well‑trodden path.

### Poll cycle (60 s)

```
t = 0    : connect Alpicool → read char → parse → disconnect
t = ~5s  : connect EcoFlow  → read char → parse → disconnect
t = 5..60: idle, display rotates pages every 3s
t = 60   : repeat
```

If a connection attempt fails, mark the device stale (grey out on display) and retry next cycle. Do not block the display rotation on a failing device.

### Display / UI

- 72×40 monochrome. Use `u8g2` (broad font support) or `Adafruit_SSD1306` (simpler). Lean toward **u8g2** for the compact bitmap fonts that make numbers readable at this size.
- Two pages, rotate every 3 s:
  - **Page 1 (fridge):** `-18°C` big, `set -20°C` small, tiny status pip (on/off, batt‑protect).
  - **Page 2 (battery):** `84%` big, `+120W / -45W` small (in/out), `4h 12m` remaining.
- If either device is stale (>2 poll cycles missed), show `— —` and a dashed border on that page.

## BLE protocol notes (to be verified against real devices)

Neither vendor publishes a BLE spec. Both have been reverse‑engineered by the community; the MVP work will start from those notes and validate against actual traffic.

### Alpicool

- Community reports the fridge advertises a service around `1234` / `FFF0` (varies by model / firmware). One characteristic is write (commands: query, set temperature, turn on/off), another is notify (responses).
- Payloads are short binary blobs, roughly `<preamble><cmd><data><checksum>`.
- Plan: sniff with `nRF Connect` from a phone first, confirm the actual service/char UUIDs on **this specific K25**, then port the parse logic.

### EcoFlow River 2

- EcoFlow's LAN protocol (protobuf over TCP) is well documented in the `hassio-ecoflow` and `ecoflow-mqtt` communities. Its BLE protocol is *less* well documented — some reports say BLE only exposes provisioning/config and full telemetry needs WiFi.
- **This is the biggest risk in the plan.** If the River 2 doesn't expose battery %, input W, output W over BLE, the EcoFlow page reduces to "connected / not connected" or requires falling back to a WiFi/MQTT bridge (out of MVP scope).
- Plan: sniff BLE traffic during a real EcoFlow app session. If the metrics we want aren't there, downgrade the EcoFlow half of the MVP and document the finding.

## Milestones

Roughly one evening each. Each milestone is a separately merge‑able PR.

- **M1 — Hello display.** Flash the board, initialize the 72×40 OLED with u8g2, show "bledash" and a pixel‑animated dot. Confirm which GPIOs the OLED I²C is on for this specific clone. Merge.
- **M2 — BLE scan.** NimBLE scan for 10 s, dump advertisers with name + MAC + RSSI to serial. Bring the C3 into range of both target devices and record what they advertise. Merge.
- **M3 — Alpicool driver.** Connect to the fridge, read the notify characteristic, parse temp + setpoint + on/off. Render it on the display. Merge.
- **M4 — EcoFlow driver.** Same for EcoFlow. If the metrics aren't available over BLE, land a stub that shows "no BLE telemetry" and document the finding. Merge.
- **M5 — Scheduler + rotation.** 60‑second poll cycle, page rotation, stale‑detection. Ship a v0.1.0 tag. Merge.
- **M6 — Car install.** Mount, run a week, note range issues / disconnect frequency. Follow‑up PRs from that.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| EcoFlow River 2 BLE doesn't expose live telemetry | Medium‑high | Sniff first, downgrade page 2 to "connected only" if confirmed. Document in `docs/protocols/ecoflow.md`. |
| BLE range in‑car is marginal | Medium | Test drive at M5 before physical install. Consider external antenna variant if needed. |
| Alpicool BLE protocol varies between firmware revisions of the K25 | Low‑medium | Version the parser; log unknown payloads. |
| 72×40 is too small to be actually useful | Low | Prototype the UI at M1 with realistic values; abort to a larger display if unreadable at a glance. |
